# Review notes - 2026-06-08

Project status: abandoned / no longer maintained after this review. The notes
below are preserved for historical context and should not be read as a plan to
continue operating the automation.

This review combines local code inspection, visible Edge test results, prior
subagent findings, and official-source research. Do not include cookies, auth
state, or account identifiers in this file.

## Official-source findings

Access date: 2026-06-08.

Microsoft official sources:

- Microsoft Services Agreement, published 2025-07-30 and effective 2025-09-30:
  `https://www.microsoft.com/en-us/servicesagreement`
- Microsoft Support - Limiting your searches in Microsoft Rewards:
  `https://support.microsoft.com/en-us/accounts-billing/rewards/limiting-your-searches-in-microsoft-rewards`
- Microsoft Support - Getting the most out of Microsoft Rewards:
  `https://support.microsoft.com/en-US/accounts-billing/rewards/getting-the-most-out-of-microsoft-rewards`
- Microsoft Support - Learn about Microsoft Rewards:
  `https://support.microsoft.com/en-us/account-billing/learn-about-microsoft-rewards-c5ab735d-c6d9-4bb9-30ad-d828e954b6a9`
- Microsoft Support - How to earn Microsoft Rewards points:
  `https://support.microsoft.com/en-us/account-billing/how-to-earn-microsoft-rewards-points-83179747-1807-7a5e-ce9d-a7c544327174`
- Microsoft Rewards official pages:
  `https://www.microsoft.com/en-us/rewards/about`
  `https://www.microsoft.com/en-us/bing/features/rewards`

Key conclusions:

- Microsoft defines a Rewards "Search" as a manually entered search by an
  individual user for good-faith personal research. Bot, macro, automated, or
  fraudulent query methods are excluded.
- Microsoft Support tells users not to use programs, bots, or macros to assist
  searches. It also warns against rapid back-to-back searches, meaningless
  strings, quickly clicking through articles, or using bookmarks to accumulate
  points quickly.
- Repeated violations may lead to account suspension and invalidated points.
- Rewards is personal and non-commercial. A person may have only one Rewards
  Account, and a household may have at most six. VPN or location-obfuscation use
  can lead to cancellation and point forfeiture.
- Microsoft may cancel the Rewards account, disqualify future participation,
  forfeit points, or cancel/suspend redemptions if it believes the account was
  tampered with, abused, or used fraudulently.
- Search point caps and offers vary by level, region, time, and activity terms.
  Offers are not permanent guarantees.

Project implication:

- The project must not claim "safe from ban".
- The project must not add evasion, bypass, high-volume search, or automation
  hardening against Microsoft enforcement.
- The only defensible posture is conservative visible-task assistance and
  diagnostics: user-triggered, low-frequency, one action at a time, explicit
  state verification, stop on failure.

## Technical-source findings

Official technical sources:

- Playwright authentication:
  `https://playwright.dev/python/docs/auth`
- Playwright network:
  `https://playwright.dev/python/docs/network`
- Playwright tracing:
  `https://playwright.dev/python/docs/trace-viewer`
- GitHub CLI release create:
  `https://cli.github.com/manual/gh_release_create`

Key conclusions:

- Playwright storage state contains browser cookies/local storage and is
  sensitive. Keep `auth_*.json` ignored.
- Prefer explicit response/state waits for protocol research. Fixed sleeps are
  useful only as conservative pacing, not as proof of completion.
- Playwright tracing/network logging is appropriate for one-card diagnostics,
  but trace artifacts must be sanitized before sharing.
- `gh release create` is the correct current CLI path for releases; use tags and
  release notes that document local verification.

## Current implementation review

Reviewed code paths:

- `discover_cards`
- `discover_dashboard_daily_cards`
- `_click_card`
- `do_explore_search`
- `read_browse_progress`
- `wait_for_credit`
- failure cache helpers
- `do_quiz`
- `trace_card`
- `main_run`

### Strengths

- Normal runs only use visible card clicks; direct href fallback is not used in
  normal task handlers.
- Default search quota, search bonus, and Copilot are off unless explicitly
  requested.
- App check-in, daily streak, and quest task groups are detected but skipped.
- `/earn` Browse on Bing progress such as `10/18` is now parsed and logged.
- Ordinary Bing `+3` search points are no longer treated as Explore-card success
  if Browse on Bing progress did not advance.
- Same-day uncredited cards are cached in `.rewards_failures.json`, preventing
  repeat clicks later the same day.
- Auth/log/failure-cache files are ignored.

### High-risk issues

1. `Card` lacks stable offer metadata.

   Multiple Explore cards share the same `ML2PCR` href. Title+href is not a
   stable task identity, and title text can change by locale or status. The next
   major improvement should parse Next.js/React Flight payloads and capture
   `offerId`, hash, completion, lock, and progress fields.

2. `wait_for_credit()` still uses discovery absence as success.

   Discovery can miss cards due to lazy loading, viewport, or transient page
   rendering. Treating one missed scan as "card removed from earn list" is a
   false-positive risk. For Explore tasks that failed to advance browse
   progress, do not accept "missing from discovery" unless a completed state for
   the same offer is explicitly read. Prefer stable offer-state comparison.

3. `wait_for_credit()` and `read_points()` navigate pages while verifying.

   This can disturb current tab/page state. It is acceptable for conservative
   post-action verification today, but should be replaced by a dedicated state
   read that does not depend on broad navigation whenever possible.

4. `trace_card(..., search=True)` is not aligned with `do_explore_search()`.

   It uses `bing_search_url()` direct navigation for Explore diagnostics, while
   the real handler types in the opened Bing page. Either align trace with the
   handler or label the current path as protocol-only.

5. `do_quiz()` may still click too broadly.

   It sorts links by parsed `WQSCORE`, which is good, but fallback locators can
   still match any `WQCI`/`WQSCORE` link by text if parsing is incomplete. Do not
   click arbitrary options. If a clear correct option is not present, fail.

6. `conservative_skip_reason()` is a tactical patch.

   Skipping zero-point activated Explore cards is based on 2026-06-08 observed
   behavior. Once offer metadata is parsed, replace this with explicit
   completed/activated/progress state handling. A better interim rule is
   failure-cache driven: allow one visible attempt if the card has not failed
   today, require browse progress, then cache and stop on no progress.

### Medium-risk issues

- `read_browse_progress()` only parses nearby text after localized labels. If
  Microsoft changes the label or layout, it returns `None` and Explore handling
  falls back to point/card-state checks.
- Failure cache keys still use title+href. This can collide or miss if titles
  change, and local clock reset may not match Rewards reset. Use offer metadata
  when available, and store before/after progress and point deltas for review.
- Search keywords are heuristic and locale-dependent. They should stay simple,
  not grow into evasion or mass-search logic.
- `read_bing_header_points()` can pick up ordinary search points. For Explore,
  this is now guarded by browse-progress requirements, but other handlers still
  depend on point deltas.
- `discover_cards()` includes zero-point Explore cards but not other zero-point
  stateful tasks. That is deliberate for now.

## Visible test observations

On 2026-06-08, visible Edge testing showed:

- Before the change, `/earn` Browse on Bing was `0/18`.
- The script discovered:
  - `app_checkin +5` (skipped)
  - `计划一次快速的短途旅行` (`explore_search`, zero points, already activated)
  - `保护重要的事物 +10`
  - `规划你的未来 +10`
  - `在 Bing 上找到优惠` (zero points)
- `保护重要的事物` advanced Browse on Bing from `0/18` to `10/18` and later
  showed `10 已完成`.
- `计划一次快速的短途旅行`, `规划你的未来`, and `在 Bing 上找到优惠` did not
  advance browse progress during testing.
- `计划一次快速的短途旅行` and `在 Bing 上找到优惠` produced ordinary `+3`
  Bing search points in some runs. The updated logic correctly did not accept
  that as Explore completion.
- Follow-up run skipped the conservative/failed-today cards and did not repeat
  failed clicks.

## Recommended next steps

1. Build sanitized fixtures.

   Save sanitized `/earn` and `/dashboard` HTML/Flight snippets without cookies
   or account IDs. Add parsing tests before another broad behavior change.

2. Parse offer metadata.

   Investigate `self.__next_f`, RSC/Flight payloads, and `POST /earn`
   `Next-Action` bodies. The goal is stable `Card` fields:
   `offer_id`, `hash`, `progress_current`, `progress_total`, `is_completed`,
   `is_locked`, `is_activated`.

3. Replace "card missing" success with explicit offer-state success.

   Accept success only when:
   - available/today points increase for non-Explore tasks;
   - browse progress increases for Explore tasks;
   - target offer state changes to completed;
   - target offer disappears because the parsed offer list no longer contains
     it and the surrounding page state is otherwise stable.

4. Improve trace tooling.

   Add a `--trace-card ... --typed-search` path that exactly mirrors
   `do_explore_search()`: click visible card, keep opened Bing tab, type search,
   collect `POST /earn`, `reportActivity`, flyout, and before/after offer state.

5. Fix documentation/behavior drift around `rnoreward=1`.

   Do not reintroduce a blanket `rnoreward=1` filter. Version `v1.1.3` allowed
   visible daily links with this parameter because a real visible daily card
   credited. Filter on visibility, locked/completed text, and verification.

6. Harden quiz.

   Only click options when the visible current card exposes a correct score.
   Add selectors for the newer `.acf-button-standard__link` quiz layout. If no
   correct option is clear, fail and stop.

7. Keep documentation honest.

   The README should continue to say that Microsoft can restrict/suspend
   accounts and that this project does not bypass enforcement or guarantee
   safety.
